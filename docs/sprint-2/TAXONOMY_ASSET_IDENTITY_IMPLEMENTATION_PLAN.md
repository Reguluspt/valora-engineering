# Implementation Plan - Sprint 2: Taxonomy + Asset Identity

This implementation plan covers data modeling, APIs, validation logic, similarity calculations, and review workflows for the Taxonomy and Asset Identity systems.

## 1. Entities to Implement

### Taxonomy Schema
1. **TaxonomyNode**: Hierarchical taxonomy group representing domains, categories, subcategories, and groups.
2. **AssetFamily**: Logical groupings of similar canonical assets.
3. **AssetDNA**: Versioned attribute schemas associated with families.
4. **AssetAttributeDefinition**: Individual technical or business attribute schemas (e.g. data types, required constraints).
5. **AssetVariant**: Variants distinguishing assets by key properties (e.g. power, size).
6. **CanonicalAssetAttributeValue** & **AssetVariantAttributeValue**: Explicit technical attribute values linked to assets or variants.

### Asset Identity Schema
1. **CanonicalAsset**: Master records of recognized assets (standard name, brand, model code).
2. **AssetAlias**: Raw variations, customer-specific names, or catalog entries linked to assets or variants.
3. **IdentityCandidate**: Suggestions generated by similarity scoring algorithms.
4. **SimilarityScore**: Score components (name token distance, brand matches) backing candidates.
5. **DuplicateCandidate**: Potential duplicates detected between canonical assets.
6. **MergeDecision**: Auditable transactions preserving lineage, aliases, and variants during asset merges.
7. **IdentityReviewItem**: Items queued for manual review.
8. **IdentityDecisionLog**: Append-only execution history.

## 2. API Endpoints Expected

### Taxonomy APIs (`/api/v1/taxonomy/...`)
- `POST /api/v1/taxonomy/nodes` (Create node)
- `POST /api/v1/taxonomy/nodes/{node_id}/submit-review` (Submit for review)
- `POST /api/v1/taxonomy/nodes/{node_id}/approve` (Activate node)
- `POST /api/v1/taxonomy/nodes/{node_id}/deprecate` (Mark deprecated)
- `POST /api/v1/taxonomy/variants` (Create variant)
- `POST /api/v1/taxonomy/variants/{variant_id}/submit-review` (Submit variant review)
- `POST /api/v1/taxonomy/variants/{variant_id}/approve` (Approve variant)

### Asset Identity APIs (`/api/v1/asset-identity/...`)
- `POST /api/v1/asset-identity/candidates/generate-bulk` (Queue similarity matcher suggestions)
- `POST /api/v1/asset-identity/candidates/batch-approve` (Execute batch approvals)
- `POST /api/v1/asset-identity/candidates/{candidate_id}/reject` (Reject matching suggestion)
- `POST /api/v1/asset-identity/assets/{asset_id}/aliases` (Attach catalog aliases)
- `POST /api/v1/asset-identity/assets/merge` (Merge duplicate canonical assets)

## 3. Permissions/RBAC References
- `taxonomy:node:create` / `taxonomy:node:update` / `taxonomy:node:approve` / `taxonomy:node:deprecate`
- `taxonomy:variant:create` / `taxonomy:variant:update` / `taxonomy:variant:approve`
- `asset_identity:suggest` / `asset_identity:batch_approve` / `asset_identity:merge` / `asset_identity:alias:create`

## 4. Suggested Acceptance Tests
- Node hierarchy validation checks.
- DNA scope enforcement (preventing variant attribute definitions in canonical schemas).
- Variant uniqueness per family.
- Optimistic locking row version checks.
- Candidate generation triggers and bulk scoring validations.
- Duplicate identification and safe merge workflows (lineage checks, alias migration).

## 5. What Is Explicitly Not Included
- Knowledge Base / Evidence matching.
- QuoteBatching and appraised pricing decisions.
- OCR/Document scanning workers.
- Production AI Prompt tuning.
